## 关键概念：三层记忆架构

这是保证多批次编码一致性的核心设计。每次给LLM的完整prompt由三层拼接而成：

```
┌─────────────────────────────────────────────┐
│ 第1层：方法论常量（永不变）                  │
│   prompt_coding_rules.txt                   │
│   - 任务定义、输出格式、编码哲学             │
├─────────────────────────────────────────────┤
│ 第2层：当前编码本快照（每批后覆盖更新）      │
│   codebook_current.md                       │
│   - 已确认的编码名+定义+纳入/排除标准+样例   │
├─────────────────────────────────────────────┤
│ 第3层：决策历史摘要（每批后追加，不删）      │
│   prompt_rolling_rules.txt                  │
│   - 关键变更记录（合并/拆分/重命名+原因）    │
└─────────────────────────────────────────────┘

完整prompt = 第1层 + 第2层 + 第3层 + 本批数据
```
## 执行策略
第一轮时，rolling和current基本是"空的占位符"——它们只是在prompt里留一个位置，告诉LLM"还没有历史积累"。三层记忆的价值在第二轮及以后才真正体现。总体规则为
- 第1层：永远有内容（方法论常量）
- 第2层和第3层：首轮为空，后续才被填充。它们的存在本身就是一种提示——告诉LLM"有没有已经确认的编码规则要遵守"

不会无脑重新发明已有概念。这就是rolling和current的价值——它们让LLM在第二轮及以后"有记忆",不会把第一轮确立好的规则抛在脑后。

## 总结

### 在第一轮:
脚本层面:rolling和current被脚本读取并拼入prompt,但因为文件内容是模板(空占位符),它们在LLM看到的prompt里显示为"(初始为空)"。
LLM行为层面:LLM看到"空的"第2层和第3层,等于得到明确信号——"我现在处于冷启动状态,没有历史约束,可以自由归纳"。这是正确的首轮行为。
研究者行为层面:首轮的关键工作发生在LLM输出结果之后——研究者必须手动把审阅过的编码写进 codebook_current.md,把重大决策写进 prompt_rolling_rules.txt。

### 在第二轮及以后:
rolling和current真正"起作用":

Current:LLM按字典查询,优先复用已有编码。新数据能匹配上就匹配,不能匹配才建议新增。
Rolling:LLM遵守历史决策。不会重新创造已合并的编码,不会把已拆分的概念合回去。

### 各层在每轮的角色：
base (第1层)Current (第2层)Rolling (第3层)
- 第一轮prompt里的内容
  - base:完整方法论
  - current:"(初始为空)"
  - rolling:"(初始为空)"
- 第一轮对LLM行为的影响
  - base: 规定格式和工作流
  - current: 告诉LLM"无历史约束"
  - rolling: 告诉LLM"无历史决策"
- 第一轮结束后谁更新
  - base: 研究者极少改
  - current:研究者必改
  - rolling:研究者有决策时才改
- 第二轮prompt里的内容
  - base: 完整方法论
  - current: 15个编码的完整定义
  - rolling: 批次0的所有重大决策
- 第二轮对LLM行为的影响
  - base: 同上
  - current: LLM优先复用,谨慎新增
  - rolling: LLM遵守历史决策
- 第二轮结束后谁更新
  - base: 不改
  - current: 追加/修改编码
  - rolling: 追加新决策

### 一个重要的现实提醒
三层机制成功的前提是:研究者必须踏实地做每批结束后的回写工作。如果偷懒,第2层永远是空的,等于每批都在重新归纳——那三层机制就失效了,编码本会乱。
这也是为什么在README里反复强调"批次间的研究者工作清单"——这个清单不长,但每项都不能跳过。
