# Prompt 工程约定 (v0.2)

> 本文档补充 README.md 中关于 LLM 对话环节的**细节性约定**。
> 研究者遇到 LLM 输出异常、编码漂移、对话冗长等问题时查阅。

## 1. 四层记忆的更新节奏

| 层 | 文件 | 更新频率 | 更新触发条件 |
|----|------|---------|-------------|
| 第1层 | `00_base_rules.txt` | 极少改 | 发现编码工艺或格式规则需要调整 |
| 第2层 | `02_codebook_current.md` | 每批后 | 研究者审阅通过LLM的编码摘要后 |
| 第3层 | `01_rolling_rules.txt` | 仅重大决策 | 合并/拆分/重命名/删除编码时 |
| 第4层 | `03_focus.txt` | 每批前 | 每批开始前（即使内容不变也要确认） |

**第2层 vs 第3层的差异**：

- 第2层是"**编码本身的定义**"（这个编码是什么）
- 第3层是"**决策的痕迹**"（为什么做了这个决定）

有些变动两者都要更新。例如合并"操作卡顿"和"响应慢"为"反馈延迟"：
- 第2层：删除前两个条目，新增"反馈延迟"条目及定义
- 第3层：追加合并记录，写明原因

**第4层的独特性**：
- 每批都要打开确认一次，**即使内容不变**
- 这个"确认动作"本身是对研究方向的提醒
- 批次之间微调聚焦（如从"户外玩法"转到"户外玩法中的作弊"）就在此文件操作

## 2. 首批 vs 后续批次的 prompt 差异

首批（batch_00）是归纳编码的"冷启动"，LLM面对空的第2层和第3层。

**首批时 LLM 看到的第2层**：

```
# 当前编码本 (第2层记忆)
**版本**：v0.0（初始状态，未开始编码）
**已处理批次**：无
## 编码列表
（初始为空，首批编码后填入）
```

LLM 从此得到明确信号："我现在冷启动，无历史约束，按 Step A→B→C→D 
四步从零开始归纳"。

**后续批次时 LLM 看到的第2层**：

```
# 当前编码本 (第2层记忆)
**版本**：v0.2
**已处理批次**：batch_00, batch_01
## 编码列表
### 现实冲突
**定义**：因工作/出差/疫情等现实条件导致无法参与户外玩法
**纳入标准**：...
**排除标准**：...
**样例引文**：...
---
### 作弊行为
...
```

LLM 从此得到信号："已有 X 个稳定编码，按 Step C'→B'→D' 简化流程
优先匹配，谨慎新增"。

## 3. 聚焦主题的分层表达

第4层 `03_focus.txt` 提供三种表达粒度，研究者按需选择：

### 3.1 大方向型

适合研究初期、主题尚未结构化时：

```
## 本次研究聚焦主题
围绕户外玩法相关体验及衍生话题进行编码。
```

**行为**：LLM 会在 Step B.3 自主识别 2-4 个主题作为组织框架。

### 3.2 子方向型

适合已有初步思路但主题结构未定型：

```
## 本次研究聚焦主题
关注户外玩法的以下几个方面：
- 空间与地域
- 作弊与治理
- 投入成本
- 公平感受
```

**行为**：LLM 使用研究者给的子方向作为主题，在每个主题下归纳编码。

### 3.3 完整框架型

适合演绎性归纳（研究者已有明确理论框架）：

```
## 编码组织框架

请将编码组织到以下主题结构中：
1. 空间结构：户外玩法的空间设计（地域分布、资源布局等）
2. 机制漏洞：户外玩法中的作弊与破坏
3. 投入成本：户外玩法对现实资源的要求
4. 公平感受：户外玩法中的相对公平性体验
5. 治理效能：围绕户外玩法的举报/封号/申诉机制
```

**行为**：LLM 严格按给定框架组织，不得擅自新增/合并主题。

## 4. 对话纠偏的话术模板

研究者在审阅LLM编码时发现问题，用以下话术高效纠偏：

### 4.1 层级问题（主题 vs 编码）

**主题被当成编码**：

> "你把主题当编码用了。'空间结构'是容器，应包含多个具体编码，
> 比如'地域集中'、'场所稀缺'、'路线单一'。请重新按主题→编码的
> 层级关系产出摘要。"

### 4.2 粒度问题

**编码过于宽泛**：

> "'操作困难'涵盖了两种不同情况：一是学习曲线陡（新手看不懂教程），
> 二是界面不直观（找不到功能入口）。请拆分为两个独立编码，
> 并重新分配当前的引文。"

**编码过于细碎**：

> "'画面好看'、'美术治愈'、'配色舒服'这三个编码讲的是同一个维度
> （视觉审美体验）。请合并为一个编码'视觉美感'。"

### 4.3 命名问题

**命名不抽象**：

> "'广州嘉年华'这个编码名是具体事件名，违反了 B.4 命名规范。
> 请抽象为分析性概念，例如'活动地域集中'。"

**命名是原文堆积**：

> "'上班出差疫情不能玩'这个编码名堆积了原文词组，违反了 B.4 规范。
> 这些表述实际指向同一概念，请统一为'现实冲突'。"

### 4.4 引文问题

**引文被截断/改写**：

> "'反馈延迟'编码下的第2条引文有省略号。格式要求是引文必须是
> 原文完整片段，不允许截断。请重新输出保留完整原文。"

**引文归错编码**：

> "'美术治愈'编码下的第5条引文'教程讲得太快了'讲的是教学节奏问题，
> 不属于视觉美感。请移出，并检查是否应归入其他编码。"

### 4.5 覆盖问题

**明显漏编**：

> "第47、62、89条评论都提到了充值相关问题，但你的编码里没有相关
> 分类。按 D.0 覆盖率检查，请判断是否需要新增编码或归为范围外。"

**范围外过度使用**：

> "你把第12、38、45条都判为范围外，但它们分别提到了'步行负担'、
> '封号申诉'、'地域差异'，都属于户外玩法话题。按 3.4 范围外判定
> 的严格性规则，请重新判定。"

### 4.6 未决项处理

**LLM 逃避判断**：

> "你把U045判为范围外但注明'不确定'。按 3.4 规则，不确定的内容
> 应标记为未决项请我裁定，不能扔进范围外。请重新分类。"

## 5. 常见LLM失误与应对

### 5.1 CSV格式错乱

**症状**：
- 输出3列或4列而不是2列
- 表头写成"编码,引文"而不是"分类,检索词"
- 引文被加了引号或省略号

**应对**：

> "你的CSV不符合格式要求：[具体列出问题]。请严格按第1层第5节
> 步骤4的要求重新输出：只有两列（分类、检索词），表头原样，
> 引文不加任何标记、不截断、前后不含标点。"

### 5.2 跳过编码工艺

**症状**：LLM 直接给出CSV，没经过阅览→确认→摘要流程；或者看起来
"顺序处理"（有多个同义编码并存）。

**应对**：新开对话，重新粘贴prompt并强调：

> "请严格按第1层第5节'工作流约定'执行。收到数据后先输出
> 【批次阅览报告】，等待我确认后再编码。"

### 5.3 忽略第2层编码本

**症状**：新创建的编码与第2层已有编码语义几乎一致。

**应对**：

> "你创建的'按钮不灵敏'，与第2层中的'反馈延迟'语义高度重叠。
> 按 C.1 定义优先匹配原则，请合并到'反馈延迟'下，除非你能说明
> 两者的实质差异。"

### 5.4 忽略第3层决策

**症状**：LLM 重新创建了历史上已被合并/删除的编码。

**应对**：

> "注意第3层决策历史第X条：'操作卡顿'和'响应慢'已在批次2合并为
> '反馈延迟'。请不要重新拆分，将相关引文统一编为'反馈延迟'。"

### 5.5 上下文溢出征兆

**症状**：回复明显不如之前细致、忽略部分数据、引文有改写迹象。

**应对**：立即开新对话窗口。**四层记忆机制的设计初衷就是支持随时开新窗口
而不丢失连续性**。旧窗口中有价值的纠偏话术可以摘抄进第3层。

## 6. gold set 比对使用

首批完成后，选 10-15 条最有代表性的评论作为 gold set。

**使用时机**：
- 每 3-4 批后跑一次
- 怀疑编码漂移时立即跑

**使用方法**：
1. 新开一个对话窗口，注入当前四层记忆
2. 只输入 gold set 的原始评论
3. 让LLM编码，得到编码结果
4. 对比首次编码结果：
   - 90%+ 一致 → 稳定，可继续
   - 70-90% 一致 → 轻度漂移，检查第2层定义是否需要加强
   - < 70% 一致 → 严重漂移，停下来重构第2层和第3层

## 7. 批次间的研究者工作清单

每批处理完成后，在进入下一批前，完成：

- [ ] 保存 LLM 的阅览报告到 `batches/batch_NN_preview.md`
- [ ] 保存 LLM 的编码CSV到 `batches/batch_NN_coding.csv`
- [ ] 更新 `knowledge/02_codebook_current.md`（第2层）
- [ ] 如有重大决策，追加 `knowledge/01_rolling_rules.txt`（第3层）
- [ ] 如聚焦主题有微调，更新 `knowledge/03_focus.txt`（第4层）
- [ ] 追加 `knowledge/prompt_log.json`（结构化变更日志）
- [ ] 运行 `05_saturation_check.py` 查看饱和度
- [ ] （可选）每3-4批跑一次 gold set 比对

**每项都不能跳过**。跳过第2层更新会导致后续批次的LLM看到过时定义，
漂移从这里开始。

## 8. 编码工艺四步法速查

复习一下 `00_base_rules.txt` 第4节的工艺：

### 首轮（第2层为空）

- **Step A** 开放抽取：通读全数据，抽取相关概念表达
- **Step B** 标准化：语义合并→边界界定→主题归属→命名规范
- **Step C** 逐条赋码：定义优先匹配，无法匹配才新增
- **Step D** 全局回溯：覆盖率→编码本自检→数据反扫→边界复核→范围外复核

### 后续（第2层非空）

- **Step C'** 优先匹配
- **Step B'** 仅对候选新编码标准化
- **Step D'** 回溯检查

研究者无需管 LLM 内部如何执行这些步骤（LLM 会在内部做），
但**审阅时应按这个结构检查**，发现跳步征兆立即纠正。
